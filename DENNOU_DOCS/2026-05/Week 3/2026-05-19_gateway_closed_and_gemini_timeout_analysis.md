## 問題1: "gateway connect failed: Error: gateway closed (1000)"

### 症状

- WebSocket 接続が突然切断される
- Close code 1000 (= 正常終了)
- クライアント側エラー: `gateway closed (1000):` / `gateway closed (1000): normal closure`
- 散発的に発生し、再現性が低い

### 原因（3つ）

| # | 原因 | ファイル | 行 | 発生条件 |
|---|------|---------|-----|---------|
| 1 | **ハンドシェイクタイムアウト** | `src/gateway/server/ws-connection.ts` | 294-303 | 認証が一定時間内に完了しない → `close()` (= code 1000) |
| 2 | **設定変更によるgateway再起動** | `src/gateway/config-reload-plan.ts` | 103 | `gateway.*` の設定変更 → `kind: "restart"` → 全接続切断 |
| 3 | **hello応答送信失敗** | `src/gateway/server/ws-connection/message-handler.ts` | 1281-1282 | WebSocket接続確立直後のhello送信に失敗 |

### 影響

- CLI / WebUI の接続が突然切れる
- 自動再接続で復旧するが、操作中だと中断になる
- 致命的ではないがユーザー体験を損ねる

### 現状の対策

DennouAibou では `session.reset.mode = "off"` にしてある（セッションリセットによる再接続は発生しない）。
ハンドシェイクタイムアウトは稀な事象なので、現時点では特に対応不要。

## 問題2: google-gemini-cli ツール実行タイムアウト

### 症状

- Gemini CLI provider 使用時にツール実行が頻繁にタイムアウト
- エラーメッセージ: `Model Fallback: openai-codex/gpt-5.4 (selected google-gemini-cli/gemini-3-pro-preview; timeout)`
- 特にコンテキストが大きい（739k tokens）状態で顕著

### 原因

| # | 要因 | ファイル | 行 | 影響度 |
|---|------|---------|-----|--------|
| 1 | **LLM idle timeout デフォルト60秒** | `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts` | 11 | 🔴 **大** — タイムアウト直撃 |
| 2 | **Large context→APIレイテンシ増大** | — | — | 🟡 **中** — 739k tokensの処理に数十秒かかる |
| 3 | **Replay sanitization + turn validation** | `src/plugin-sdk/provider-replay-helpers.ts` | 136 | 🟡 **中** — 毎ターン履歴全体を走査 |
| 4 | **SSRFガードのDNS+Agent使い捨て** | `src/infra/net/ssrf.ts` | — | 🟢 **小** — さっきのcache修正で改善済み |

### フローチャート

```
ツール実行要求
  ↓
DNS解決 + TLSハンドシェイク ← さっきのcache修正で改善
  ↓
Gemini APIリクエスト送信
  ↓
Geminiが739k tokensのコンテキストを処理（数十秒）
  ↓
Geminiがツール結果を返す
  ↓
Replay sanitization + turn validation を実行（数百ms）
  ↓
次のツール呼び出し or 完了
```

60秒のタイムアウトは、特に large context でギリギリの設定。

### 対策

**即効性:**
- `agents.defaults.llm.idleTimeoutSeconds` を 120〜180 に伸ばす

**根本的:**
- コンテキストサイズを減らす（pruneが効くツール出力がある場合のみ）
- 軽いモデルに切り替える

## 追記: 2026-05-19 — PR #49751 cherry-pick

### 対応内容

上流 PR #49751 (fix: clear WS handshake timer early, increase timeouts) を `[SYNC]` で取り込み。

- **ハンドシェイクタイムアウトを 3秒 → 12秒に延長**
- `clearHandshakeTimer()` を認証完了後から接続リクエスト検証直後に移動
- Sashakhar1 の `67e41722e6` をベースに、`DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS` だけ 12,000ms に変更
- 残余の2コミット（review fixup）はスキップ

### 同時に修正した問題

- デバイス認証パスで変数 `canvasCapability` / `nextClient` が重複宣言されるバグを修正（デバイス認証側の変数を `device*` にリネーム）
- `DEFAULT_RESET_MODE` を `"daily"` → `"off"` に変更（`[FIX-SOUL]`）

### KASOU デプロイ後の追加 hotfix

PR #49751 の conflict 解決時に、WebUI の `connect` ハンドシェイク経路を壊すミスが混入した。
ユーザー視点では、OpenClaw Gateway ダッシュボードは表示されるが、赤い「接続」ボタンを押しても中へ入れない状態になった。
玄関までは見えているのに、鍵を回した瞬間に内側のドアが閉まるような症状。

#### 原因

| # | 原因 | ファイル | 影響 |
|---|------|---------|------|
| 1 | `setClient()` 登録が二重化していた | `src/gateway/server/ws-connection/message-handler.ts` | 2回目の登録経路で `hello-ok` 送信前に詰まる |
| 2 | `setClient()` は `void` なのに `if (!setClient(...))` で判定していた | `src/gateway/server/ws-connection/message-handler.ts` | 常に失敗扱いになり、全接続を中断する |
| 3 | `endpoint` / `peerLabel` の未定義参照が残っていた | `src/gateway/server/ws-connection/message-handler.ts` | タイムアウト時に `ReferenceError` で gateway 全体が落ちる |

#### 修正

- 早すぎる `nextClient` 登録を削除し、`deviceNextClient` の single-registration path に戻した。
- `setClient(deviceNextClient)` を直接呼び、`void` 戻り値を真偽値として扱わないようにした。
- `endpoint` を `endpointLabel`、`peerLabel` を `clientLabel` に置き換えた。
- `clearConnectAuthTimer()` / `clearHandshakeTimer()` は、認証成功後に1回だけ呼ぶ形へ整理した。
